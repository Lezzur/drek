## 1. Problem Statement

Rick is a solo AI automation consultant who creates two types of videos: (1) Loom cover letter videos demonstrating specific technical capabilities to hiring managers, and (2) YouTube tutorials targeting potential clients who want AI systems built. Both require significant pre-production work — choosing which projects to feature, which features to highlight, writing full scripts, and planning shot composition. This planning is currently unassisted, costing hours per video and producing inconsistent quality.

Prospect Intelligence generates a growing volume of job listings requiring video demonstrations. Manual planning does not scale. Rick is also launching a YouTube channel as a client acquisition channel, which requires a repeatable planning and scripting pipeline. Neurocore's projects layer — just shipped — provides the project knowledge that makes intelligent, automated video planning feasible for the first time.

**DREK solves the pre-production problem.** It takes a requirement (from a PI listing or manual topic input), matches it to the right projects in Rick's portfolio via Neurocore, generates a structured video plan with scene cards, writes complete spoken-word scripts in Rick's voice, and exports shoot instructions he can follow while recording.

## 2. Users

### Primary User: Rick

- **Role:** Solo AI automation consultant. Builds agentic systems, RAG pipelines, internal tools.
- **Context of use:** Rick has just ingested a job listing in PI that requires a video demonstration, OR he is planning a YouTube video on a specific AI automation topic. He needs to quickly plan what to show, how, and in what order — then get a full script he can follow while recording.
- **Success:** Video planning and scripting takes minutes instead of hours. Plans are specific enough and scripts polished enough that Rick can open Loom, follow the script, and record in one or two takes.
- **Failure:** Plans are generic, scripts sound robotic, system doesn't understand his actual projects or their demo-ready features, requiring him to redo the work manually.

### YouTube Audience Model (not a user of DREK, but shapes output)

- **Primary audience:** Potential clients — business owners, founders, ops leads who want AI systems and automations built. They watch Rick's videos to evaluate whether he can build what they need.
- **Secondary audience:** Practitioners and aspiring builders. DREK does not optimize for this group, but content remains technically credible.
- **Default orientation:** Client-buyer perspective unless Rick explicitly overrides per-video.

### Anti-Users

DREK is not for video editors, general content creators, or teams. It is a single-user tool optimized for Rick's project portfolio and workflow.

## 4. Features

### 4.1 Listing Ingestion via Neurocore Signal Polling

**What:** DREK polls Neurocore's pending-video endpoint on a configurable cron to detect new PI-ingested job listings that require video demonstrations.

**Behavior:**
- DREK runs a background cron job (configurable interval, default: every 30 minutes)
- Each poll calls Neurocore's PI signal consumption endpoint (Gap 5 in Neurocore gap spec)
- Neurocore returns a list of PI listings not yet consumed by DREK. This includes both listings explicitly flagged as requiring video demonstrations and all other new listings.
- For listings flagged as requiring video, DREK creates a pending plan entry with status `awaiting_review`
- All other listings appear in a separate "available listings" view where Rick can manually select any listing to create a cover letter plan for — even if the listing doesn't explicitly require video
- DREK marks consumed signals as acknowledged to Neurocore so they are not returned on subsequent polls

**States:**
- **Polling active:** Cron is running, last poll timestamp displayed in UI header
- **Polling paused:** Cron disabled via config or UI toggle. No new listings ingested.
- **Manual poll in progress:** Rick clicked "Check now." Immediate poll executing outside the cron cycle.
- **Poll error:** Neurocore unreachable or returned an error. UI shows last successful poll time and error indicator. DREK retries on next cron cycle.
- **No new listings:** Poll returned zero results. No action taken. Last poll time updated.
- **New listings found:** Pending plan entries created. Count displayed as notification badge.

**Priority:** P0 — Must have

---

### 4.2 Video Requirement Detection

**What:** Given a PI job listing (via Neurocore signal), DREK analyzes the listing requirements to identify what needs to be demonstrated on video.

**Behavior:**
- Extracts explicit video requirements from listing text (e.g., "show automation work relating to lead pipelines")
- Identifies implicit requirements from job description skills and qualifications
- Produces a structured requirement summary: skills to demonstrate, tools/technologies to show, time constraints if specified, any specific deliverables mentioned
- Requirement summary is stored on the plan and displayed to Rick for review before planning proceeds

**States:**
- **Analyzing:** LLM is processing the listing. Spinner displayed.
- **Analysis complete:** Requirement summary displayed. Rick can edit, add, or remove detected requirements before proceeding.
- **Analysis failed:** LLM returned an error or unusable output. Error message shown. Rick can retry or enter requirements manually.

**Priority:** P0 — Must have

---

### 4.3 Project-to-Requirement Matching

**What:** DREK queries Neurocore's project registry to find which of Rick's projects best demonstrate the detected requirements, then ranks and presents matches.

**Behavior:**
- Sends detected requirements + Neurocore context query to the LLM
- LLM receives full project data from Neurocore: project name, category tags, demonstrable features (with interface type, demo-readiness flag, and narrative metadata — Gap 3), tech stack, status
- LLM returns ranked list of project matches with: project name, matched features, relevance explanation, suggested demo sequence
- Rick reviews matches and can accept, reject individual projects, reorder, or add projects manually
- For cover letter mode: typically 1-2 projects matching specific requirements
- For YouTube mode: may include 2-4 projects illustrating a broader topic

**States:**
- **Matching:** LLM is processing. Progress indicator shown.
- **Matches found:** Ranked project list displayed with relevance scores. Rick can accept/reject/reorder.
- **No matches:** No projects match the requirements. Rick is prompted to either refine requirements or select projects manually.
- **Partial matches:** Some requirements matched, others unmatched. Unmatched requirements highlighted. Rick decides whether to proceed with gaps or adjust.
- **Match accepted:** Rick has confirmed the project selection. Plan moves to scene generation.

**Priority:** P0 — Must have

### 4.4 Two Planning Modes

**What:** DREK supports two distinct planning modes with fundamentally different composition rules, pacing, tone, and success criteria. These are separate planning profiles, not parameterized variants of one template.

#### 4.4.1 Cover Letter Mode

**Audience:** Hiring managers and recruiters (evaluative)
**Target runtime:** <2 minutes default, adjustable via target runtime input (D-21)
**Composition rules:**
- Open with a brief, confident intro stating Rick's relevant experience (5-10 seconds)
- Demonstrate exact skills requested — no tangential showcasing
- Each demo point is concise: show the thing, explain what it does, move on
- No storytelling structure, no personality flourishes, no jokes
- Close with a clear call to action (availability, next steps)
- Tone: professional, direct, trust-building
- Pacing: brisk but not rushed — every second earns its place
- Typical scene count: 3-5 scenes

**Success criteria:** Hiring manager watches the full video, sees Rick can do what the job requires, and moves forward with the application.

#### 4.4.2 YouTube Mode

**Audience:** Primary — potential clients wanting AI systems built. Secondary — practitioners.
**Target runtime:** 8-15 minutes default, adjustable via target runtime input (D-21)
**Composition rules:**
- Storytelling structure: problem → cost/pain → solution → result
- Open with a hook that frames the business problem (not the technology)
- Lead with business outcomes, cost savings, and operational improvements
- Show how the system works from the user/client perspective first, then technical details
- Rick's personality and spoken voice carry the delivery — warmth, authority, occasional humor
- Include "proof moments" — real data, real interfaces, real results
- Close with what this means for the viewer's business + channel CTA
- Tone: authoritative but approachable, client-facing
- Pacing: deliberate, with room for emphasis and pauses
- Typical scene count: 8-12 scenes

**Success criteria:** Potential client finishes the video thinking "I want this person to build something like this for my business." Practitioners find it technically credible.

**Behavior:**
- Mode is selected when creating a new plan (cover letter or YouTube)
- Mode determines which composition rules the LLM uses for scene generation and script writing
- Mode defaults can be overridden per-plan via user constraints
- Both modes share the same data pipeline (Neurocore query, project matching) but diverge at scene generation

**Priority:** P0 — Must have

### 4.5 Full Script Writing

**What:** DREK generates complete spoken-word scripts per scene in Rick's voice — not outlines, bullet points, or notes.

**Behavior:**
- After scene cards are generated, DREK writes a full script for each scene
- Scripts are written in Rick's spoken voice using Neurocore's spoken-voice profile (Gap 4)
- Each script includes:
  - Complete spoken text (word-for-word what Rick will say)
  - Emphasis cues (words or phrases to stress, marked inline)
  - Pacing notes (where to pause, where to speed up)
  - Transition phrases between scenes (how to bridge from one scene to the next)
- Scripts respect the target runtime: total script word count calibrated to ~150 words/minute spoken pace, adjusted per mode
- Cover letter scripts are tighter and more direct; YouTube scripts allow more personality and storytelling
- Rick can edit any script inline in the scene card UI

**States:**
- **Generating:** LLM is writing scripts. Per-scene progress shown.
- **Complete:** All scene scripts generated. Total estimated runtime displayed.
- **Runtime mismatch:** Generated scripts exceed or undershoot target runtime by >15%. Warning displayed with suggestion to add/remove scenes or trim/expand scripts.
- **Edit mode:** Rick is editing a scene script inline. Changes auto-save.

**Priority:** P0 — Must have

---

### 4.6 Scene Card Generation

**What:** DREK generates structured shot-by-shot scene cards for the video plan.

**Behavior:**
- Each scene card contains:
  - **Scene number** (ordinal position)
  - **Scene title** (short descriptive label)
  - **Description** (what happens in this scene — what's being shown, explained, demonstrated)
  - **Framing notes** (e.g., "screenshare, terminal visible", "screenshare, browser with app dashboard", "headshot, speaking to camera")
  - **Script** (complete spoken-word text for this scene — see 4.5)
  - **Estimated duration** (in seconds, based on script word count and pacing)
  - **Project reference** (which Neurocore project this scene demonstrates, if applicable)
  - **`storyboardImageUrl`** (nullable — reserved for v2 image generation, always `null` in v1)
- Scene cards are generated by the LLM based on: planning mode composition rules, matched projects, detected requirements, target runtime, user constraints
- Scene count is calibrated to target runtime: cover letter typically 3-5, YouTube typically 8-12
- Scenes are ordered by the LLM's composition logic per mode

**States:**
- **Generating:** LLM is creating scene cards. Spinner displayed.
- **Generated:** All scene cards displayed in order. Total estimated runtime shown.
- **Empty:** No scenes generated (LLM error or edge case). Error message shown. Rick can retry.

**Priority:** P0 — Must have

---

### 4.7 Target Runtime Input

**What:** Rick specifies the exact target runtime for any video plan. This is a first-class input field, not just a mode default.

**Behavior:**
- Every new plan has a target runtime field
- Mode defaults pre-fill the field: cover letter = 120 seconds, YouTube = 600 seconds (10 minutes)
- Rick can override the default to any value
- DREK uses target runtime to calibrate:
  - Number of scenes generated
  - Script density (words per scene)
  - Level of detail per demonstration point
  - Pacing notes in scripts
- After scene generation, DREK displays estimated runtime vs. target runtime
- If estimated runtime deviates from target by >15%, a warning is shown with suggestions

**Input:**
- Numeric field in seconds, with a display helper showing minutes:seconds
- Minimum: 30 seconds
- Maximum: 3600 seconds (60 minutes)

**Priority:** P0 — Must have

---

### 4.8 User Constraints

**What:** Rick provides per-plan constraints that modify how DREK generates scenes and scripts.

**Behavior:**
- Free-text input field on the plan creation form
- Rick enters any constraints that should influence the plan (examples below)
- Constraints are passed to the LLM alongside mode rules, requirements, and project data
- Constraints override mode defaults where applicable

**Example constraints:**
- "Headless only — no GUI demos"
- "Screenshare only — no talking head"
- "Focus on the RAG pipeline, skip the frontend"
- "Frame this for a technical audience, not client-buyer"
- "Keep it under 90 seconds"
- "Include a comparison with manual process"
- "Mention pricing at the end"

**States:**
- **Empty:** No constraints entered. LLM uses mode defaults only.
- **Populated:** Constraints displayed on plan. LLM incorporates them.

**Priority:** P1 — Should have

### 4.9 Scene Cards UI

**What:** Web interface for reviewing, reordering, and editing scene cards and scripts.

**Technology:** Hono 4.x server-rendered HTML + HTMX for interactivity (D-6)

**Behavior:**
- Scene cards displayed as a vertical list, one card per scene
- Each card shows: scene number, title, description, framing notes, script (collapsible), estimated duration, project reference
- **Reorder:** Move-up and move-down buttons on each card. HTMX swaps the card positions without full page reload. Scene numbers update automatically.
- **Inline edit:** Click on any text field (title, description, framing notes, script) to enter edit mode. Text field becomes a textarea. Changes auto-save on blur via HTMX PATCH request.
- **Delete scene:** Button on each card. Confirmation prompt before deletion. Remaining scenes renumber.
- **Add scene:** Button at bottom of the list and between existing scenes. Opens a blank scene card in edit mode.
- **Collapse/expand scripts:** Toggle to show/hide the full script text per card. Default: scripts visible.
- **Runtime bar:** Visual indicator at the top showing estimated total runtime vs. target runtime. Green if within 15%, yellow if 15-30% deviation, red if >30%.

**Layout:**
- Desktop only (Chrome). No mobile responsive design (D-16).
- Full-width card layout. No sidebar.
- System fonts, functional/utilitarian styling (D-17).
- Monochrome with minimal accent color for interactive elements.

**States:**
- **Loading:** Plan is being fetched. Skeleton cards displayed.
- **Viewing:** Scene cards displayed. All fields readable. Edit/reorder/delete controls visible.
- **Editing:** One field on one card is in edit mode (textarea). Other cards remain in view mode.
- **Reordering:** Card is being moved. Brief animation or visual feedback on swap.
- **Empty plan:** No scenes exist (plan was just created or all scenes deleted). Prompt to generate scenes or add manually.
- **Saving:** HTMX request in flight. Subtle indicator on the affected card.
- **Save error:** HTMX request failed. Error indicator on the card. Retry option.

**Priority:** P0 — Must have

### 4.10 Exportable Shoot Instructions

**What:** A single document that Rick references while recording. Combines scene cards and scripts into a printable/readable format optimized for the recording session.

**Behavior:**
- Generated from the finalized plan (after Rick has reviewed, reordered, and edited scenes)
- Document contains:
  - Video title and type (cover letter / YouTube)
  - Target runtime
  - Target audience reminder
  - For each scene, in order:
    - Scene number and title
    - Framing notes (what should be on screen)
    - Full script text (formatted for easy reading — larger font consideration, clear paragraph breaks)
    - Transition note to next scene
    - Estimated duration
  - Total estimated runtime at the bottom
- Export formats: HTML page (printable via browser print), plain text (copyable)
- Accessible via "Export shoot instructions" button on the plan page

**States:**
- **Generating:** Document being assembled. Brief spinner.
- **Ready:** Document displayed in a new view/tab. Print and copy buttons available.
- **Stale:** Plan has been modified since last export. Warning shown. Re-export button available.

**Priority:** P0 — Must have

- All subsequent steps (project matching, scene generation, script writing) proceed identically to listing-triggered plans

**States:**
- **Input form:** Empty form displayed. Topic is required.

- **Validation error:** Topic field empty. Inline error shown.

**Priority:** P0 — Must have

### 4.12 LLM Provider Abstraction

**What:** Interface pattern with two implementations — `ClaudeCLIProvider` and `CodexCLIProvider` — allowing Rick to switch LLM providers via configuration.

**Behavior:**
- `LLMProvider` interface defines the contract: accepts a prompt (system + user messages), returns structured text output
- `ClaudeCLIProvider` invokes Claude CLI via `child_process.spawn`
- `CodexCLIProvider` invokes Codex CLI via `child_process.spawn`
- Active provider selected by environment variable or config file flag
- Provider selection is application-wide (not per-request)
- All LLM calls in DREK (requirement detection, project matching, scene generation, script writing) go through the provider interface

**Priority:** P0 — Must have

### 4.13 Plan Dashboard

**What:** Landing page showing all video plans — pending, in-progress, and completed.

**Behavior:**
- Lists all plans in reverse chronological order
- Each plan entry shows: title, type (cover letter/YouTube), status, target runtime, creation date, number of scenes
- Filters: by type (cover letter / YouTube / all), by status (pending / in-progress / complete / all)
- Click a plan to open the scene card view
- "New cover letter plan" and "New YouTube plan" buttons
- "Check now" button triggers an immediate Neurocore poll outside the cron cycle

**States:**
- **Loading:** Dashboard skeleton displayed.
- **Plans listed:** Plans displayed with status indicators.

## 5. User Flows

### 5.1 Cover Letter Flow (Listing-Triggered)

1. Rick ingests a job listing in PI that requires a video demonstration
2. PI signals Neurocore. Neurocore flags the listing for DREK consumption.
3. DREK's polling cron picks up the new listing on next cycle
4. DREK creates a pending plan (`awaiting_review`) and displays it on the dashboard with a notification badge
5. Rick opens the pending plan. Sees the listing title, company, and raw text.
6. Rick clicks "Analyze requirements." DREK extracts video requirements from the listing via LLM.
7. Requirement summary displayed. Rick reviews, edits if needed, confirms.
8. Rick clicks "Match projects." DREK queries Neurocore and presents ranked project matches.
9. Rick reviews matches. Accepts, rejects, or reorders projects. Confirms selection.
10. Rick sets target runtime (default: 120 seconds). Enters any user constraints.
11. Rick clicks "Generate plan." DREK generates scene cards and scripts per cover letter mode composition rules.
12. Scene cards displayed. Rick reviews, reorders (move-up/down), edits scripts inline.
13. Runtime bar shows estimated vs. target runtime. Rick adjusts if needed.
14. Rick clicks "Export shoot instructions." Document generated.
15. Rick opens Loom, follows the shoot instructions, records the video.

### 5.2 YouTube Flow (Manual Topic)

1. Rick clicks "New YouTube plan" on the dashboard
2. Enters topic, optional description/angle, target runtime (default: 600s), optional constraints
3. DREK queries Neurocore for projects relevant to the topic via LLM
4. Project matches displayed. Rick reviews, confirms selection.
5. Rick clicks "Generate plan." DREK generates scene cards and scripts per YouTube mode composition rules.
6. Scene cards displayed. Rick reviews, reorders, edits scripts inline.
7. Runtime bar shows estimated vs. target runtime. Rick adjusts if needed.
8. Rick clicks "Export shoot instructions." Document generated.
9. Rick records the video following the instructions.

### 5.3 Batch Cover Letter Flow

1. Rick ingests multiple job listings in PI during a batch session
2. Rick clicks "Check now" on the DREK dashboard to trigger an immediate poll (not waiting for cron)
3. DREK polls Neurocore. Listings explicitly requiring video become pending plans automatically. All other listings appear in the "available listings" view.
4. Rick reviews pending plans (auto-created from video-required listings). He can dismiss any he doesn't want to plan for.
5. Rick browses available listings and manually selects any additional listings he wants to create cover letter videos for.
6. Rick works through selected plans sequentially: analyze → match → generate → review → export. Each cover letter requires per-listing judgment — no batch-generate.
7. Rick records all cover letter videos in a single recording session using the exported shoot instructions

## 6. Neurocore Integration Contract

DREK depends on Neurocore for all project knowledge and PI signal routing. Five prerequisite changes (gaps) must be completed in Neurocore before DREK can function at full capability.

### 6.1 API Calls DREK Makes to Neurocore

| Endpoint | Purpose | Gap dependency |
|----------|---------|----------------|
| `POST /v1/memory/context` | Query project data for requirement matching — retrieves projects with features, demo-readiness, tech stack, status | Gap 1 (DREK task type in injection profiles), Gap 3 (narrative/demo metadata) |
| Pending-video signal endpoint (new) | Poll for PI listings flagged as requiring video demonstrations | Gap 5 (PI signal consumption endpoint) |
| Signal acknowledgment (new) | Mark consumed signals so they are not returned on subsequent polls | Gap 5 |
| `POST /v1/memory/context` (voice profile) | Retrieve Rick's spoken-voice profile for script generation | Gap 4 (spoken-voice profile) |
| `POST /v1/memory/context` (identity) | Retrieve Rick's professional identity, experience summary, positioning | Existing — no gap |

### 6.2 Neurocore Gaps Summary

| Gap | Description | Estimated effort | Blocks |
|-----|-------------|-----------------|--------|
| 1 | DREK-specific task types in injection profiles | ~30 minutes | Project matching quality |
| 2 | Project-status temporal cron | ~2-4 hours | Stale project data |
| 3 | Narrative/demo-readiness metadata in crawl schema | ~1-2 hours | Feature-level matching |
| 4 | Spoken-voice profile for script generation | ~2-3 hours | Script voice fidelity |
| 5 | PI signal consumption endpoint | ~1-2 hours | Listing-triggered mode entirely |

**Dependency order:** Gap 5 must be done before DREK's polling can function. Gaps 1, 3, and 4 affect quality but DREK can technically operate (with degraded output) without them. Gap 2 is independent background maintenance.

### 6.3 Graceful Degradation

If Neurocore is unreachable:
- Listing-triggered mode is unavailable. Dashboard shows "Neurocore unreachable — polling paused."
- Manual topic input (YouTube mode) still works but project matching is disabled. Rick must specify projects manually.
- Script generation falls back to generic voice (no spoken-voice profile). Warning shown: "Voice profile unavailable — scripts may not match your spoken style."

## 8. Composition Rules Reference

These rules are used as system-level instructions when generating scene cards and scripts. They are not user-facing — they drive LLM prompt construction.

### 8.1 Cover Letter Composition Rules

```
MODE: cover_letter
AUDIENCE: Hiring managers and recruiters (evaluative)
TONE: Professional, direct, trust-building
PACING: Brisk but not rushed — every second earns its place

STRUCTURE:
1. Opening (5-10s): Brief, confident intro. State relevant experience. No fluff.
2. Demonstrations (bulk of video): One scene per skill/requirement. Show the thing, explain what it does, move on. No tangential showcasing.
3. Closing (5-10s): Clear call to action — availability, next steps, contact.

RULES:
- Demonstrate exact skills requested. Nothing extra.
- No storytelling structure.
- No personality flourishes, humor, or casual asides.
- Each demo point is concise: show → explain → next.
- Total runtime must respect target. Cut scope before cutting quality.

ANTI-PATTERNS:
- Generic introductions ("Hi, I'm excited about this role...")
- Demonstrating skills not in the listing
- Spending time on project backstory
- Ending without a clear next step
```

### 8.2 YouTube Composition Rules

```
MODE: youtube
AUDIENCE_PRIMARY: Potential clients — business owners, founders, ops leads wanting AI systems built
AUDIENCE_SECONDARY: Practitioners and aspiring builders (don't optimize for, don't repel)
TONE: Authoritative but approachable, client-facing, Rick's personality carries
PACING: Deliberate, with room for emphasis and pauses

STRUCTURE:
1. Hook (15-30s): Frame the business problem. NOT the technology. Make the viewer feel the pain.
2. Problem/cost (60-90s): What this costs a business today. Manual process, time waste, error rate, opportunity cost.
3. Solution walkthrough (bulk): Show how the system works. Client/user perspective FIRST, then technical details. Include "proof moments" — real data, real interfaces, real results.
4. Results (60-90s): What changed. Numbers if available. Business impact.
5. Closing (15-30s): What this means for the viewer's business. Channel CTA.

RULES:
- Lead with business outcomes, not technology.
- Frame showcases as "here's what this does for a business" not "here's how I built this."
- Rick's spoken voice and personality carry the delivery — warmth, authority, occasional humor.
- Include real artifacts: dashboards, outputs, metrics.
- Technically credible enough that practitioners respect it.

ANTI-PATTERNS:
- Opening with "Hey guys, today we're going to..."
- Leading with tech stack or architecture
- Talking about the build process before showing the result
- Generic advice without concrete examples from Rick's portfolio
- Ending without connecting back to the viewer's business needs
```

### 9.2 Reliability

- DREK should handle Neurocore being temporarily unreachable without crashing (graceful degradation per 6.3)
- LLM provider errors should surface to the user with a retry option, not silent failure
- Plan data is persisted to Firestore after each step — no data loss on browser close or server restart
- Polling cron should survive server restarts (pm2 manages process lifecycle)

### 9.3 Security

- No authentication for v1 (D-13). Single user, localhost access only.
- Firestore access via service account key stored as environment variable, not committed to repo
- No user-supplied data is executed as code. All LLM prompts are constructed server-side.

### 9.4 Browser Support

Desktop Chrome only (D-16). No mobile, no other browsers tested.

### 9.5 UI/UX

- System fonts (D-17). No custom typography.
- Functional/utilitarian styling. No brand assets, no logo.
- Monochrome palette with a single accent color for interactive elements (buttons, links, active states).
- Readable text: minimum 16px body text, 14px for secondary/metadata text.
- Sufficient contrast for readability (not formally WCAG-audited, but no light-gray-on-white).

## 12. Assumptions and Risks

### Assumptions

| Assumption | Impact if wrong | Fallback |
|------------|----------------|----------|
| Neurocore's project data is granular enough for quality matching | Project suggestions will be irrelevant, requiring manual override every time | Rick manually selects projects. Neurocore enrichment work (Gaps 1, 3) improves data quality. |
| ~150 wpm is a reasonable spoken pace estimate for runtime calibration | Runtime estimates will consistently overshoot or undershoot | Make wpm configurable per mode. Calibrate after first 5 videos. |
| Rick's spoken voice can be captured well enough in a text profile for script generation | Scripts will sound generic/robotic, defeating the purpose of full script writing | Voice profile bootstraps from written voice profile, then calibrates iteratively from Rick's script edits and approvals. No manual transcript seeding — Rick doesn't have existing spoken transcripts to feed. |
| Cover letter videos under 2 minutes are effective for hiring managers | Rick may need longer videos for complex roles | Target runtime is adjustable per-plan — the 2-minute default is a starting point. |
| Batch polling (every 30 minutes) is responsive enough for Rick's workflow | Rick may want to trigger planning immediately after ingesting a listing | Add a manual "check now" button that triggers an immediate poll. |

### Risks

Refer to Discovery Brief Section 7 for the full risk table. Key risks restated:

1. **Project matching quality** (Medium likelihood, High impact) — Mitigated by Neurocore gap work and Rick's review step before generation.
2. **Mode blending** (Low likelihood, High impact) — Mitigated by separate composition rule profiles, not parameterized templates.
3. **Script voice fidelity** (Medium likelihood, High impact) — Bootstraps from written voice profile (Gap 4). Calibrates iteratively as Rick edits and approves scripts. No manual transcript seeding available.
